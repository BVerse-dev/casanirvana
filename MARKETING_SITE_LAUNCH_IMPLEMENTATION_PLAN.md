# Casa Nirvana Marketing Site Launch Implementation Plan

## Objective

Launch the Casa Nirvana marketing website as a Next.js application that preserves the approved WordPress/Saliver presentation, removes WordPress as a production dependency, accurately represents every Casa Nirvana application, and passes documented release gates.

The captured WordPress output is the visual specification. Work after capture is a controlled hardening and content migration, not a redesign.

## Current Baseline

- Status date: 2026-07-19.
- Next.js production preview: `http://localhost:3001`.
- WordPress comparison source: `http://localhost:8882` while WordPress Studio is running.
- Captured documents: 107.
- Mirrored runtime assets: 636, with zero capture failures.
- Evidence manifest: `apps/marketing-web/public/wordpress-snapshot/manifest.json`.
- Source defect: `/our-services/` returns HTTP 404 in WordPress.
- Marketing production build: passing.
- Native Next.js interfaces retained: `/api/contact` and `/api/onboarding`.

## Non-Negotiable Controls

- Do not redesign approved WordPress sections during parity work.
- Do not regenerate from WordPress after the baseline is approved. The repository snapshot becomes the editable source.
- Do not combine fidelity fixes, content rewrites, route deletion, and infrastructure changes in one change set.
- Do not mark a route complete without evidence at all required viewports.
- Do not expose API keys, WordPress credentials, Studio files, SQL exports, or private Supabase credentials.
- Do not retain WordPress admin, editor, REST, AJAX, cron, or WooCommerce server dependencies in production.
- Do not remove demo or shop routes until the final route inventory and redirects are approved.
- Preserve native Next.js API routes and production application links throughout the migration.

## Execution Sequence

## Stage 1 - Freeze the fidelity baseline

Status: In progress.

### Work

- Preserve the generated manifest, route HTML, assets, source timestamp, and database backup reference.
- Record screenshots for WordPress and Next.js at 1440x900, 1280x800, 1024x768, 768x1024, 390x844, and 360x800.
- Compare every P0 route for header, menus, section order, section height, text wrapping, image crop, spacing, typography, colors, shadows, borders, responsive stacking, footer, and animation states.
- Record approved differences explicitly; do not accept an undocumented approximation.
- Capture navigation-open, accordion-open, carousel, hover/focus, pending, success, failure, and reduced-motion states where applicable.

### Acceptance gate

- Every P0 route has comparison evidence at six viewports.
- No missing local asset, unintended horizontal overflow, broken internal link, or unexplained browser console error remains.
- Desktop and mobile navigation match the WordPress behavior.
- Any intentional difference has owner approval in the parity tracker.

## Stage 2 - Harden the static Next.js runtime

Status: Pending Stage 1 evidence.

### Work

- Classify every loaded script and stylesheet as required presentation runtime, replaceable interaction runtime, or removable WordPress/editor runtime.
- Remove admin-bar markup/styles, editor tooling, Bertha, WordPress admin links, REST discovery, XML-RPC, feeds, oEmbed, cron, and other control-plane references.
- Remove live `localhost:8882`, WordPress REST, `admin-ajax.php`, and plugin endpoint dependencies.
- Keep only the CSS, fonts, media, icons, and animation scripts required by approved public routes.
- Add a deterministic repository command that rebuilds the static route manifest from the committed snapshot without contacting WordPress.
- Add link and asset checks that operate against the Next.js output.
- Preserve the original DOM/class contract until parity is signed off.

### Acceptance gate

- The marketing application works with WordPress Studio stopped.
- Network inspection contains no request to WordPress, `localhost:8882`, `wp-admin`, `wp-json`, `admin-ajax.php`, or external demo services.
- Required header, menu, carousel, accordion, scroll, and animation behavior still works.
- Production build remains passing.

## Stage 3 - Replace forms and dynamic public behavior

Status: Pending Stage 2 runtime inventory.

### Work

- Preserve the exact visible Contact Form 7 and onboarding styling.
- Submit onboarding through same-origin `POST /api/onboarding`, then forward to the existing backend onboarding contract with its key attached server-side.
- Submit general enquiries through same-origin `POST /api/contact`, then forward to the protected backend contact endpoint.
- Support accessible labels, keyboard operation, inline validation, honeypots, payload limits, duplicate-submit prevention, rate limiting, and truthful pending/success/failure states.
- Replace WordPress search, cart, wishlist, comments, and account behavior only if those routes are approved for launch; otherwise remove or redirect them in Stage 5.
- Ensure production app calls-to-action target the correct resident, guard, manager, and administration destinations.

### Acceptance gate

- No browser-delivered secret or `NEXT_PUBLIC_*` server credential exists.
- Contact and onboarding happy paths and failure paths are evidenced.
- Duplicate submissions, invalid payloads, unavailable backend, and email-delivery failure produce controlled responses.
- Forms work with WordPress stopped.

## Stage 4 - Build the Casa Nirvana content source of truth

Status: Pending parity approval; planning may begin earlier.

### Work

- Audit resident mobile, guard mobile, facility-management, superadmin, marketplace, billing, visitor, security, communication, and community capabilities from the application repositories and current product records.
- Create a claims matrix: feature, supported application, intended audience, evidence/source, approved wording, CTA, and destination.
- Define Casa Nirvana voice, terminology, capitalization, prohibited claims, trust language, and Ghana/location wording.
- Convert editable text, links, repeated labels, FAQs, pricing data, and SEO fields into typed repository content.
- Bind typed content to stable captured section/component slots without changing approved structure, spacing, or visual behavior.
- Replace demo copy route by route in this order: Home, Our Products, Residents, Security Guards, Facility Managers, Marketplace, Pricing, About, FAQs, Contact, Legal.
- Record content approval separately from visual parity approval.

### Acceptance gate

- Every product claim is traceable to implemented functionality or explicitly marked upcoming.
- No Saliver, grazing, generic agency, demo SaaS, or unrelated WooCommerce copy remains on P0 routes.
- All CTAs lead to valid production or approved prelaunch destinations.
- Content changes do not cause unapproved layout drift at the six required viewports.

## Stage 6 - SEO, accessibility, privacy, and performance

Status: Pending stable routes and content.

### Work

- Remove localhost, WordPress, demo-domain, and duplicate canonical metadata.

### Acceptance gate

- Accessibility review has no unresolved launch-blocking issue.

- Agreed performance budgets pass on representative desktop and mobile profiles.

## Stage 7 - Deployment, cutover, and rollback

Status: Pending all prior P0 gates.

### Work

- Create or confirm a dedicated Vercel project rooted at `apps/marketing-web`.
- Configure preview and production values for backend URL, server-only form keys, canonical site URL, analytics, and approved application destinations.
- Run the production build and launch smoke matrix in preview.
- Record WordPress URL, final export, database backup, DNS state, and previous hosting target as rollback artifacts.
- Complete domain and DNS cutover only after P0 signoff.
- Keep WordPress non-public but recoverable through the rollback window.
- Monitor forms, backend delivery, 404s, redirects, analytics, and client errors after cutover.

### Acceptance gate

- Every P0 route, CTA, form, metadata endpoint, redirect, and backend health check passes on the production domain.
- Rollback steps are tested and have named ownership.
- WordPress is archived only after the agreed stability window.
